# Herotech: Architecture, Design Choices & Concurrency Notes

This document highlights the core engineering decisions, architectural trade-offs, and design patterns implemented in the Dragon Market project. The primary focus here isn't just about routing HTTP requests; it's about ensuring absolute financial consistency under concurrent write streams while keeping the codebase clean and highly testable.

## 1. Storage & Database Choice (Postgres 18 + UUID v7)

Instead of relying on eventually consistent systems or introducing a cache-heavy layer early on, **PostgreSQL** was chosen as the single source of truth. Financial transactions, spending limits, and live auction updates require strict ACID guarantees.

- **UUID v7 Over v4/Serial:** We rejected auto-incrementing integer IDs to prevent exposing database cardinality via the public API. However, traditional random UUID v4 causes severe B-Tree index fragmentation as the dataset grows, dropping write performance. We opted for **UUID v7**, which uses a chronological 48-bit timestamp prefix. This ensures time-ordered/sequential inserts (append-only style on disk) while maintaining the global uniqueness of a UUID.
- **Why Postgres 18?** Version 18 supports native `uuidv7()` generation out of the box. This keeps our database migration scripts completely clean, removing the need for custom `PL/pgSQL` functions or application-side ID generation boilerplate.
- **Non-Unique Item Names:** Item names are intentionally not marked as `UNIQUE`. In an fantasy loop, multiple players can discover or craft identical loot (e.g., "Iron Broadsword"). They are uniquely isolated by their primary identifiers.

## 2. Refactoring the Item Marketplace Lifecycle

The original requirement implied merging item generation and marketplace exposure into a single step. I found this structurally suboptimal and split the logic into two decoupled phases:

1. **Minting to Private Vault (`CreateItem`):** Fresh items drop directly into a guild's private inventory ledger with a status of `sold` and a `NULL` list price.
2. **Market Exposure (`ListForSale`):** The owner can subsequently expose common or rare items to the public catalog by defining a fixed listing price, shifting the state to `available`.

**Legendary Constraints:** Legendary tier items cannot be assigned a fixed price or sold directly. They are explicitly blocked at the service boundary and can only change hands via the active live auction platform.

## 3. Financial Safety, Quotas & Concurrency Boundaries

Handling concurrent purchases and bid placements safely without risking double-selling or negative wallet balances was a major focus.

- **Deterministic Lock Ordering:** To eliminate the risk of database deadlocks during intersecting multi-row updates, we enforce a strict **Asset-before-Guild** locking sequence (`SELECT ... FOR UPDATE`) inside the repository layer:
  - **Direct Purchase:** Locks the Market Item row first, then the Buyer's Guild row.
  - **Bidding Flow:** Locks the Auction row first, then the Bidder's Guild row.
- **Quota Race Mitigation:** By locking the target Guild row immediately at the start of any financial valuation, concurrent requests from the exact same guild are instantly serialized. This prevents edge cases where a user might attempt to bypass their daily spending limit via parallel API bursts.
- **Derived Balances:** Reserved bid capital is never duplicated across redundant database columns to prevent state drift. Instead, a guild's `available_balance` is calculated dynamically by subtracting active, non-leading bid constraints from their core token balance. I used explicit database type casting (`::BIGINT`) inside the query definitions to force `sqlc` to generate native Go `int64` types instead of ambiguous empty interfaces.
- **Dropping `CancelBid`:** The `CancelBid` endpoint and its underlying mechanics were completely deprecated. Allowing players to speculatively bid up prices to lock out other buyers and then freely retract their stakes introduces high potential for market manipulation. Submitted bids are immutable escrow locks until outbid or finalized.

## 4. Decoupled Clean Architecture & Safe Error Cascading

The application strictly follows a `Handler -> Service -> Repository` layering pattern to isolate transport mechanics from database drivers.

- **Transaction Encapsulation:** Multi-step transactional blocks are entirely encapsulated inside their respective repository concrete methods (`PlaceBidTransaction`, `PurchaseLimitOrder`) rather than driving them from the Service layer. This keeps our Service interfaces 100% database-agnostic and free of `pgx.Tx` boilerplate without needing a complex Unit of Work abstraction.
- **Data Domain Quarantine:** Database-specific structs generated by `sqlc` are restricted to the infrastructure layer. The mapping layer (`converters.go`) translates them into clean, domain-valid entities using standard Go primitives and native pointers, shielding business logic from driver null-type constraints.
- **Sanitizing Internal 500 Failures:** Database index violations (such as trying to launch duplicate concurrent auctions on a single item via `idx_one_active_auction_per_item`) are caught directly in the repository. We catch these constraint tokens, map them to explicit domain errors like `ErrItemAlreadyInAuction`, and allow the handler to output a standard `400 Bad Request` instead of leaking internal database structures through unhandled 500 error responses.
- **SELECT \* Rationale:** Our core queries واکشی complete states (`SELECT *`) because validation checks (like wallet thresholds, ownership, item types) require evaluating the complete historical and state context of an asset. If a specific partial layout is needed later, `sqlc` handles this natively via standalone inline row structures.

## 5. Pragmatic Infrastructure Choices (Why No Redis or RabbitMQ?)

We deliberately avoided external distributed layers like Redis or message queues to keep local setups lean and reduce multi-container complexity.

- **Auction Expiry via Background Worker:** Instead of using Redis TTL callbacks or complex cron daemons, auction settlement is handled asynchronously by a localized, in-process thread (`AuctionWorker`) powered by Go's native `time.Ticker`. It queries candidates periodically and processes them using deterministic database row-locks.
- **Why Not Lazy Expiry?** We rejected lazy settlement (evaluating if an auction is expired only when a user calls its specific view route). In a proper financial pipeline, sellers should get their gold and losing bids should be released automatically when the timer expires, without depending on a future unrelated HTTP request to wake up the system.
- **Oracle Resilience Wrapper:** The external price service integration is defined as a simple interface with a fault-tolerant structural wrapper (`SafePriceOracle`). If the remote HTTP provider or mock registry crashes or experiences downtime, the local decorator gracefully falls back to a secure `100 Gold` baseline memory-cached fallback to keep the marketplace listing flow fully functional.

## 6. Resolving Ambiguities & Scope Extensions

Where the challenge brief lacked clear instructions, I prioritized end-to-end testability over assumptions:

- Added dedicated endpoints for **Guild Creation**, **Wallet Summaries**, and **Manual Funding Deposits** (`POST /guilds/{id}/wallet/deposit`). Without these utility lifelines, verifying daily transaction limits, spending thresholds, and wallet balance bounds in a local environment would be impossible.
- The bidding endpoint was moved from an item scope to an auction scope (`POST /auctions/{id}/bid`). Bids belong to the live auction lifecycle, not the standalone asset profile.

## Future Roadmap (What I'd Add Next)

1. **Transactional Outbox Pattern:** Move the background worker to an event-driven model. Writing finalization metrics into a local `outbox` table during the same DB transaction ensures reliable downstream notifications via brokers like Kafka without risking dual-write failures.
2. **Distributed Cache for Public Feeds:** Introduce Redis specifically to cache massive read streams like `GET /items` (Marketplace storefront) to lower database load, keeping Postgres row-level locks strictly focused on intensive write modifications.
3. **Idempotency Key Middleware:** Inject an idempotency tracking mechanism using unique request signatures (`X-Idempotency-Key`) to securely protect write gateways (like buying or bidding) against accidental network double-submissions.
